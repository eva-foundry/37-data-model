## 1. Quick Start

```powershell
# Navigate to the model root
Set-Location C:\AICOE\eva-foundation\37-data-model

# Load the assembled model into a variable
$m = Get-Content model/eva-model.json | ConvertFrom-Json

# Confirm all 11 layers are populated
$m.PSObject.Properties.Name | Sort-Object
# agents, containers, endpoints, feature_flags, infrastructure,
# literals, personas, requirements, schemas, screens, services

# You're ready. No source files needed.
```

## 2. Loading the Model

The assembled file `model/eva-model.json` is generated by `assemble-model.ps1`.
It merges all 11 layer files into a single queryable object.

**Never hand-edit `eva-model.json`.** Edit the layer file, then re-assemble.

```powershell
# After editing any layer file:
.\scripts\assemble-model.ps1

# Validate cross-references:
.\scripts\validate-model.ps1   # must exit 0
```

## 3. Query Examples by Audience

### 3.1 Developer — Exploring the API surface

```powershell
$m = Get-Content model/eva-model.json | ConvertFrom-Json

# All endpoints with their auth requirements
$m.endpoints | Select-Object id, method, path, auth, status | Format-Table

# Only implemented endpoints
$m.endpoints | Where-Object { $_.status -eq 'implemented' } | Select-Object id, path

# What does POST /v1/chat accept and return?
$ep = $m.endpoints | Where-Object { $_.id -eq 'POST /v1/chat' }
$ep.request_schema
$ep.response_schema
$ep.auth
$ep.feature_flag

# Which personas can call POST /v1/ingest/upload?
($m.endpoints | Where-Object { $_.id -eq 'POST /v1/ingest/upload' }).auth

# Which endpoints write to the 'jobs' container?
$m.endpoints | Where-Object { $_.cosmos_writes -contains 'jobs' } | Select-Object id
```

### 3.2 Architect — Understanding the data model

```powershell
$m = Get-Content model/eva-model.json | ConvertFrom-Json

# All Cosmos containers with partition keys and TTL
$m.containers | Select-Object id, partition_key, ttl_seconds | Format-Table

# Fields in the 'sessions' container
($m.containers | Where-Object { $_.id -eq 'sessions' }).fields | Format-Table

# Which endpoints read from multiple containers?
$m.endpoints | Where-Object { $_.cosmos_reads.Count -gt 1 } |
  Select-Object id, cosmos_reads | Format-List

# Infrastructure: all provisioned Azure resources
$m.infrastructure | Where-Object { $_.status -eq 'provisioned' } |
  Select-Object id, type, azure_resource_name | Format-Table

# What env var maps to Azure Search?
($m.infrastructure | Where-Object { $_.type -eq 'azure_search_index' }).env_var
```

### 3.3 Frontend Developer — Screen and literal queries

```powershell
$m = Get-Content model/eva-model.json | ConvertFrom-Json

# All screens in admin-face
$m.screens | Where-Object { $_.app -eq 'admin-face' } |
  Select-Object id, route, status | Format-Table

# All API calls TranslationsPage makes
($m.screens | Where-Object { $_.id -eq 'TranslationsPage' }).api_calls

# All literals used by TranslationsPage
$m.literals | Where-Object { $_.screens -contains 'TranslationsPage' } |
  Select-Object key, default_en, default_fr | Format-Table

# Find all literals missing a French translation
$m.literals | Where-Object { -not $_.default_fr -or $_.default_fr -eq '' } |
  Select-Object key, default_en

# How many literals per screen?
$m.screens | ForEach-Object {
  $sid = $_.id
  $count = ($m.literals | Where-Object { $_.screens -contains $sid }).Count
  [PSCustomObject]@{ screen = $sid; literals = $count }
} | Format-Table
```

### 3.4 Project Manager — Requirement traceability

```powershell
$m = Get-Content model/eva-model.json | ConvertFrom-Json

# All epics and their status
$m.requirements | Where-Object { $_.type -eq 'epic' } |
  Select-Object id, title, status, sprint | Format-Table

# Requirements with no test coverage (action needed)
$m.requirements | Where-Object { $_.test_ids.Count -eq 0 } |
  Select-Object id, title, type, status | Format-Table

# Requirements that are planned but have no satisfied_by items
$m.requirements | Where-Object { $_.status -eq 'planned' -and $_.satisfied_by.Count -eq 0 } |
  Select-Object id, title

# Full traceability: which requirement covers POST /v1/chat?
$m.requirements | Where-Object { $_.satisfied_by -contains 'POST /v1/chat' }

# Stories for a given persona
$m.requirements | Where-Object { $_.personas -contains 'admin' } |
  Select-Object id, title, type, status | Format-Table
```

### 3.5 DevOps / Infrastructure — Azure resource inventory

```powershell
$m = Get-Content model/eva-model.json | ConvertFrom-Json

# Full Azure resource inventory
$m.infrastructure | Select-Object id, type, azure_resource_name, status | Format-Table

# All Key Vault secrets that need to be created
$m.infrastructure | Where-Object { $_.keyvault_secret_name } |
  Select-Object id, keyvault_secret_name, env_var | Format-Table

# Resources still in 'planned' state (not yet provisioned)
$m.infrastructure | Where-Object { $_.status -eq 'planned' } |
  Select-Object id, type, azure_resource_name | Format-Table

# All env vars needed by eva-brain-api
$m.infrastructure | Where-Object { $_.service -eq 'eva-brain-api' -and $_.env_var } |
  Select-Object env_var, azure_resource_name | Sort-Object env_var | Format-Table

# Cosmos containers with their parent account
$cosmos = $m.infrastructure | Where-Object { $_.type -like 'cosmos_*' }
$cosmos | Select-Object id, type, azure_resource_name | Format-Table
```

## 4. Agent Skills Playbook

This section defines **exactly how an AI agent (Copilot, Claude, custom skill)
should use the model at each task type**. These are the canonical patterns.

---

### Skill: New Feature Implementation

An agent implementing a new endpoint or screen should follow this sequence:

```
1. Load model
2. Query: does this endpoint already exist?
3. Query: which container does it read/write?
4. Query: what persona is permitted?
5. Query: what feature flag gates it?
6. Implement
7. Update model/endpoints.json + model/schemas.json
8. Re-assemble + validate
```

**PowerShell pre-flight (copy into agent context before coding):**
```powershell
$m = Get-Content model/eva-model.json | ConvertFrom-Json

# Does the endpoint already exist?
$m.endpoints | Where-Object { $_.path -like '*translations*' } | Select-Object id, status

# What does the 'translations' container look like?
$m.containers | Where-Object { $_.id -eq 'translations' } | Select-Object -ExpandProperty fields

# What feature flag is needed?
$m.feature_flags | Where-Object { $_.id -like '*translation*' }
```

---

### Skill: Bug Triage — "Something is broken in X screen"

```powershell
$m = Get-Content model/eva-model.json | ConvertFrom-Json

# What endpoints does the broken screen call?
$calls = ($m.screens | Where-Object { $_.id -eq 'TranslationsPage' }).api_calls

# What auth does each endpoint require?
$m.endpoints | Where-Object { $_.id -in $calls } | Select-Object id, auth, feature_flag

# What containers do those endpoints touch?
$m.endpoints | Where-Object { $_.id -in $calls } |
  Select-Object id, cosmos_reads, cosmos_writes | Format-List

# Are any of those endpoints 'stub' or 'planned' (not implemented)?
$m.endpoints | Where-Object { $_.id -in $calls -and $_.status -ne 'implemented' }
```

**Typical finding:** A screen calls a stub endpoint → tells the agent exactly where
to look in the source, without grep.

---

### Skill: Refactoring — Rename a Cosmos field

Before touching a single source file, run the impact analysis:

```powershell
$m = Get-Content model/eva-model.json | ConvertFrom-Json
$container = 'translations'

# 1. Which endpoints touch this container?
$eps = $m.endpoints | Where-Object {
  $_.cosmos_reads -contains $container -or $_.cosmos_writes -contains $container
}
"Affected endpoints: $($eps.Count)"
$eps | Select-Object id, status

# 2. Which request/response schemas reference those endpoints?
$schema_ids = $eps | ForEach-Object { $_.request_schema, $_.response_schema } |
  Where-Object { $_ } | Sort-Object -Unique
$m.schemas | Where-Object { $_.id -in $schema_ids } | Select-Object id, kind

# 3. Which screens call those endpoints?
$screens = $m.screens | Where-Object {
  ($_.api_calls | ForEach-Object { $_ -in $eps.id }) -contains $true
}
"Affected screens: $($screens.Count)"
$screens | Select-Object id, route

# 4. Which literals are displayed by those screens?
$screen_ids = $screens.id
$m.literals | Where-Object {
  ($_.screens | ForEach-Object { $_ -in $screen_ids }) -contains $true
} | Select-Object key, default_en | Sort-Object key
```

This is the "Siebel field rename" in modern form — complete blast radius in
under 10 seconds, zero source file reads.

### Skill: Localization Quality Check

```powershell
$m = Get-Content model/eva-model.json | ConvertFrom-Json

# All literals with both EN and FR populated
$m.literals | Where-Object { $_.default_en -and $_.default_fr } | Measure-Object | Select-Object Count

# Literals missing a French translation (actionable list for translators)
$m.literals | Where-Object { -not $_.default_fr -or $_.default_fr -eq '' } |
  Select-Object key, default_en | Format-Table

# Namespace breakdown (find untranslated clusters)
$m.literals | ForEach-Object {
  $ns  = $_.key.Split('.')[0]
  $has = if ($_.default_fr) { 'has-fr' } else { 'missing-fr' }
  [PSCustomObject]@{ namespace = $ns; fr_status = $has }
} | Group-Object namespace, fr_status | Select-Object Name, Count | Format-Table
```

## 5. Cross-Layer Impact Analysis

The `scripts/impact-analysis.ps1` script automates the most common chain query.

```powershell
# Impact of renaming a field in a container
.\scripts\impact-analysis.ps1 -field key -container translations

# Impact of changing an endpoint
.\scripts\impact-analysis.ps1 -endpoint "GET /v1/translations"

# Impact on a screen
.\scripts\impact-analysis.ps1 -screen TranslationsPage
```

Expected output structure:
```
Impact on field 'key' in container 'translations'
  Endpoints:
    GET  /v1/config/translations/{language}   [implemented]
    POST /v1/admin/translations               [stub]
    ...
  Schemas:
    TranslationEntry (response · model)
    TranslationUpdate (request)
  Screens:
    TranslationsPage  /admin/translations  [implemented]
  Literals:
    translations.key_label
    translations.key_placeholder
    ...
```

## 6. Updating the Model

### Adding a new endpoint

1. Edit `model/endpoints.json` — add entry conforming to `schema/endpoint.schema.json`
2. If new response shape: edit `model/schemas.json`
3. If new Cosmos read/write: verify container id exists in `model/containers.json`
4. Run: `.\scripts\assemble-model.ps1`
5. Run: `.\scripts\validate-model.ps1`  ← must exit 0

### Adding a new screen

1. Edit `model/screens.json` — every `api_calls` entry must be a real endpoint id
2. Edit `model/literals.json` — add an entry per string key; `screens` array must include the new screen id
3. Assemble + validate

### Adding a new Azure resource

1. Edit `model/infrastructure.json` — type must be one of the schema enum values
2. Assemble + validate

### Never do this

```powershell
# WRONG — hand-editing the assembled file
notepad model/eva-model.json

# RIGHT — edit the layer file, then assemble
notepad model/endpoints.json
.\scripts\assemble-model.ps1
```

## 8. Common Mistakes

| Mistake | Consequence | Fix |
|---------|-------------|-----|
| Adding endpoint to model with `cosmos_reads: []` when it actually reads a container | Silent gap in impact analysis | Read the route handler; fill in actual containers |
| Using a screen name that doesn't match the screen `id` in screens.json | Validator reports dangling literal | Match `screens` array values exactly to screen `id` values |
| Setting a requirement's `satisfied_by` to `["EPIC-001"]` | Validator rejects — epics are not endpoints or screens | `satisfied_by` must contain endpoint ids or screen ids only |
| Editing `eva-model.json` directly | Next `assemble-model.ps1` overwrites your edit silently | Always edit layer files |
| Forgetting to add a new agent to `agents.json` | Model says 3 agents; reality is 4 | Same-PR rule: update model when adding agent |
| Marking endpoint `status: implemented` before it is wired | Sprint planning queries are wrong | Use `stub` until the route handler is complete and tested |
